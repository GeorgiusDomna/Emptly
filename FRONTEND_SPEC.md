# Спецификация фронтенда: приватный чат (готовый к продакшену)

## 1. Цель фронтенд-части

Фронтенд должен обеспечивать надежный и быстрый пользовательский сценарий приватного чата по ссылке:
- создание комнаты и получение ссылки;
- переход по ссылке и вход в комнату;
- обмен сообщениями в реальном времени;
- корректная обработка ошибок и разрывов соединения;
- отсутствие утечек сообщений между комнатами.

---

## 2. Рекомендуемый стек для готового проекта

## 2.1 Базовый слой
- **React 19 + TypeScript**
- **Vite** (выбран для этого проекта, с плановой миграцией с webpack)
- **React Router v7** для маршрутов:
  - `/`
  - `/connect-room`
  - `/room/:roomId`

## 2.2 Интерфейс и стили
- **Tailwind CSS v4**
- **shadcn/ui + Radix UI** (сохранить текущий подход)
- **clsx + tailwind-merge** (уже используется)
- Подход к стилям: utility-first + точечные SASS-модули только для сложных раскладок

## 2.3 Данные и состояние
- **TanStack Query** для HTTP-операций (health, предварительные проверки комнаты, будущие API)
- **Zustand** для локального состояния сессии чата (userId, roomId, состояние соединения, UI-флаги)
- Сообщения хранятся только в состоянии активной страницы комнаты

## 2.4 Транспорт и протокол
- **Нативный WebSocket API** + обертка в хук `useChatSocket`
- **zod** для рантайм-валидации входящих и исходящих событий
- Стратегия переподключения: экспоненциальная задержка + джиттер + верхний лимит попыток

## 2.5 Надежность и наблюдаемость
- **Sentry (frontend SDK)** для ошибок рантайма
- Метрики Web Vitals (через плагин Vite или собственную отправку)
- Централизованный `ErrorBoundary`

## 2.6 Качество и тестирование
- **ESLint + typescript-eslint + eslint-plugin-react-hooks**
- **Prettier**
- **Vitest + React Testing Library + MSW**
- **Playwright** для e2e (минимум сценарий двух пользователей в комнате)

## 3. Архитектура фронтенда

## 3.1 Структура каталогов

Рекомендуемая структура:
- `src/app` — инициализация приложения, роутер, провайдеры
- `src/pages` — компоненты уровня страниц
- `src/features/chat` — сокет, доменные типы, конвейер сообщений
- `src/features/room` — создание комнаты, подключение, копирование ссылки
- `src/shared/ui` — дизайн-система
- `src/shared/lib` — утилиты, валидаторы, форматтеры
- `src/shared/config` — конфигурация окружения и константы

## 3.2 Контракты домена
- События разделены на:
  - клиент -> сервер
  - сервер -> клиент
- Каждое событие валидируется через `zod` до обработки.
- Единая модель сообщения:
  - `messageId`
  - `roomId`
  - `senderId`
  - `text`
  - `sentAt`
  - `kind` (`user` | `system`)

## 3.3 Обязательные состояния интерфейса
- `idle`
- `connecting`
- `connected_waiting_peer`
- `connected_ready`
- `room_full`
- `disconnected_reconnecting`
- `disconnected_failed`

---

## 4. Нефункциональные требования фронтенда

- Не отображать сообщения из чужих комнат.
- Не отправлять пустые и слишком длинные сообщения.
- Безопасно обрабатывать поврежденные JSON-пакеты с сервера.
- Не хранить секреты в клиентском коде.
- Время интерактивности стартовой страницы: менее 2 секунд на среднем устройстве.

---

## 5. Переменные окружения и конфигурация

- `VITE_WS_URL=wss://<domain>/ws`
- `VITE_SENTRY_DSN=<dsn>`
- `VITE_APP_ENV=development|staging|production`

Все публичные переменные должны иметь префикс `VITE_`.

---

## 6. Критерии готовности (Definition of Done)

Фронтенд считается готовым к релизу, если:
- Стабильно работает сценарий "создать комнату -> отправить ссылку -> подключить второго пользователя".
- Ошибки `room_full`, разрыв соединения и переподключение корректно отображаются в интерфейсе.
- Все события проходят рантайм-валидацию.
- Пройдены unit/integration/e2e тесты.
- После smoke-теста нет критических и major-ошибок в Sentry.
