# Спецификация бэкенда: приватный чат (готовый к продакшену)

## 1. Цель бэкенд-части

Бэкенд должен обеспечивать безопасный и предсказуемый слой общения в реальном времени для приватных комнат:
- строгая изоляция комнат;
- лимит 2 участника на комнату;
- отсутствие персистентного хранения сообщений;
- очистка памяти после завершения сессии;
- наблюдаемость и операционная надежность.

---

## 2. Рекомендуемый стек для готового проекта

## 2.1 Базовый слой
- **Node.js 20 LTS**
- **TypeScript**
- **Express** (выбран для этого проекта)
- **ws** для WebSocket транспорта

## 2.2 Архитектурные компоненты
- **RoomManager (в памяти)**: `Map<roomId, RoomState>`
- **ConnectionRegistry**: связь `socketId -> roomId/userId`
- **Слой протокола**: типизация и валидация WS-событий
- **Сервисный слой приложения**: оркестрация join/leave/send message

## 2.3 Безопасность
- **helmet**
- **cors** (строгий список разрешенных origin)
- лимиты на размер payload и ограничение частоты запросов для входящих сообщений
- серверные таймауты и heartbeat (`ping/pong`)
- никакой секретной строки подключения в репозитории, только `.env`

## 2.4 Логирование и наблюдаемость
- **pino** структурированные JSON-логи
- request-id / correlation-id в логах
- **OpenTelemetry** (минимум traces + metrics) или эквивалент
- endpoint-ы `/health` и `/ready`

## 2.5 Тесты и качество
- **Vitest** + **supertest**
- unit-тесты ключевой логики комнат
- integration-тесты WS-протокола
- нагрузочный/дымовой тест на 100+ одновременных комнат (минимум)
- ESLint + Prettier + строгая конфигурация TypeScript

## 2.6 Деплой и инфраструктура
- **Docker** (многоэтапная сборка)
- Обратный прокси (Nginx/Caddy/Ingress) с поддержкой WebSocket upgrade
- Горизонтальное масштабирование через sticky sessions на MVP-этапе,
  либо переход на общий адаптер/pub-sub (Redis) при маршрутизации чата между несколькими инстансами

## 3. Архитектура бэкенда

## 3.1 Модель данных в памяти
- `RoomState`:
  - `roomId: string`
  - `clients: Set<ClientConnection>` (максимум 2)
  - `createdAt: number`
  - `lastActivityAt: number`
- `ClientConnection`:
  - `socket`
  - `userId`
  - `joinedAt`

## 3.2 Протокол событий (контракт)

Клиент -> сервер:
- `join_room` `{ type, roomId, userId }`
- `chat_message` `{ type, roomId, userId, messageId, text, sentAt }`
- `leave_room` `{ type, roomId, userId }`

Сервер -> клиент:
- `room_joined`
- `peer_joined`
- `new_message`
- `peer_left`
- `room_full`
- `error`

Все входящие payload валидируются, иначе отправляется `error` и выполняется мягкая обработка без падения процесса.

## 3.3 Основные бизнес-правила
- Комната создается по требованию при первом `join_room`.
- В комнате максимум 2 клиента.
- Сообщения роутятся только внутри `roomId`.
- При отключении клиента:
  - убрать клиента из комнаты;
  - отправить `peer_left` оставшемуся участнику;
  - удалить комнату, если она пуста.
- TTL для пустых/неактивных комнат для защиты от утечек памяти.

---

## 4. Нефункциональные требования бэкенда

- P95 задержки доставки сообщения в комнате: целевое значение < 150 мс (в одной зоне).
- 0 случаев кросс-роум утечек сообщений.
- корректный graceful shutdown без потери процесса и без зависших сокетов.
- контроль утечек памяти при длительных soak-тестах.

---

## 5. Env и секреты

Обязательные переменные:
- `NODE_ENV`
- `PORT`
- `CORS_ORIGIN`
- `LOG_LEVEL`
- `ROOM_IDLE_TTL_MS`
- `MAX_MESSAGE_BYTES`

Опциональные:
- `OTEL_EXPORTER_ENDPOINT`
- `SENTRY_DSN`

---

## 6. Критерии готовности (Definition of Done)

Бэкенд готов к релизу, если:
- Протокол реализован и покрыт тестами (позитивные и ошибочные сценарии).
- Лимит 2 участника и изоляция комнат подтверждены интеграционными тестами.
- Нет хранения сообщений вне памяти.
- Логи, health/readiness и graceful shutdown работают.
- Секреты отсутствуют в репозитории, `.env.example` актуален.
