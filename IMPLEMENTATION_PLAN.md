# План реализации: приватный чат

## 1. Целевая архитектура

## 1.1 Компоненты
- **Фронтенд SPA** (React + TypeScript)
  - страницы: `Start`, `ConnectRoom`, `Room`
  - хук `useChatSocket` для связи в реальном времени
  - слой интерфейса и доменная логика событий
- **Бэкенд API + WS-шлюз** (Node.js + TypeScript)
  - HTTP endpoint-ы: `/health`, `/ready`
  - WS endpoint: `/ws`
  - сервисы: `RoomManager`, `ProtocolValidator`, `ConnectionRegistry`

## 1.2 Поток данных
1. Пользователь A создает комнату и получает `roomId` со ссылкой.
2. Пользователь A подключается по WS и отправляет `join_room`.
3. Пользователь B открывает ссылку и отправляет `join_room`.
4. Сервер проверяет вместимость комнаты:
   - если 2/2 занято -> `room_full`;
   - иначе `room_joined` и `peer_joined`.
5. `chat_message` маршрутизируется только клиентам конкретной комнаты.
6. При `close` сокета сервер выполняет `leave` и очищает пустые комнаты.

## 2. План реализации по этапам

## Этап 0: Подготовка репозитория

Цель: убрать технический шум и подготовить чистую основу.

Задачи:
- Удалить или изолировать устаревший код, не участвующий в MVP.
- Привести frontend и backend к понятной структуре `src`.
- Подключить единые инструменты качества:
  - ESLint
  - Prettier
  - TypeScript strict mode
- Добавить верхнеуровневый `README` и `.env.example`.

Результат:
- Единые стандарты разработки и минимальный техдолг перед стартом.

---

## Этап 1: Основа бэкенда

Цель: запустить надежный WS-бэкенд с комнатами в памяти.

Задачи:
- Реализовать `RoomManager`:
  - создание/получение комнаты;
  - вход с ограничением вместимости (2 участника);
  - выход и очистка при отключении.
- Реализовать WS-протокол и валидацию событий через `zod`.
- Добавить обработку ошибок:
  - невалидный JSON;
  - неизвестный тип события;
  - несоответствие `roomId`;
  - переполненная комната.
- Реализовать heartbeat (`ping/pong`) и idle TTL для комнат.

Результат:
- Рабочий серверный контур чата, соответствующий продуктовой модели.

---

## Этап 2: Основа фронтенда

Цель: собрать стабильный пользовательский сценарий комнат.

Задачи:
- `StartPage`:
  - генерация `roomId`;
  - формирование ссылки;
  - копирование ссылки в буфер обмена.
- `ConnectRoom`:
  - разбор ссылки или roomId из поля;
  - валидация и переход в `/room/:roomId`.
- `RoomPage`:
  - подключение через `useChatSocket`;
  - отправка и получение сообщений;
  - состояния: ожидание собеседника, комната занята, переподключение.

Результат:
- Полный путь пользователя от создания комнаты до обмена сообщениями.

---

## Этап 3: Интеграция и согласованность домена

Цель: синхронизировать контракт фронтенда и бэкенда.

Задачи:
- Вынести общую схему событий или строго синхронизировать дубликаты.
- Зафиксировать версию протокола.
- Добавить тест-кейсы совместимости:
  - join -> peer_joined;
  - chat_message -> new_message;
  - disconnect -> peer_left;
  - третий пользователь -> room_full.

Результат:
- Предсказуемая работа без скрытых конфликтов формата сообщений.

---

## Этап 4: Тестирование

Цель: подтвердить стабильность MVP.

Задачи бэкенда:
- unit-тесты для `RoomManager`;
- integration-тесты WS-сценариев;
- soak-тесты на утечки памяти при сериях подключений/отключений.

Задачи фронтенда:
- unit-тесты хука `useChatSocket`;
- компонентные тесты формы подключения и отправки сообщений;
- e2e (Playwright): два пользователя в одной комнате и изоляция от третьего.

Результат:
- Достаточный уровень test confidence для релиза MVP.

---

## Этап 5: Подготовка к продакшену

Цель: сделать проект готовым к боевому запуску.

Задачи:
- Контейнеризировать frontend и backend (Docker).
- Настроить обратный прокси и WS upgrade.
- Настроить CI pipeline:
  - lint
  - typecheck
  - test
  - build
- Подключить наблюдаемость:
  - структурированные логи;
  - отслеживание ошибок;
  - health/readiness probes.
- Выполнить security-checklist:
  - все секреты только через env;
  - строгий CORS whitelist;
  - лимиты payload и частоты запросов.

Результат:
- Репозиторий и сервисы готовы к деплою в staging/production.

## 3. Предлагаемая структура директорий

## 3.1 Бэкенд
- `server/src/app` — инициализация приложения
- `server/src/config` — конфигурация окружения
- `server/src/ws` — WS-сервер, обработчики, протокол
- `server/src/domain/rooms` — менеджер и модель комнат
- `server/src/shared` — утилиты, ошибки, логирование
- `server/tests` — unit/integration тесты

## 3.2 Фронтенд
- `client/src/app` — провайдеры и роутер
- `client/src/pages` — страницы маршрутов
- `client/src/features/room` — создание и подключение комнаты
- `client/src/features/chat` — сокет и обработка сообщений
- `client/src/shared/ui` — интерфейсные компоненты
- `client/src/shared/lib` — валидаторы и вспомогательные функции
- `client/tests` — unit/component/e2e утилиты

---

## 4. Ключевые архитектурные правила

- Единственный источник истины по участникам комнаты — бэкенд.
- Сообщения не хранятся вне оперативной памяти.
- Любое входящее событие валидируется до бизнес-логики.
- UI не должен зависеть напрямую от необработанного сокет-пакета.
- Каждое состояние соединения должно иметь понятное отображение в интерфейсе.

---

## 5. Вехи и ориентировочные сроки

- **M1 (1-2 дня):** чистка и базовый бэкенд.
- **M2 (1-2 дня):** пользовательский поток фронтенда и интеграция сокета.
- **M3 (1 день):** тесты критических сценариев.
- **M4 (1 день):** hardening, CI и готовность к деплою.

Итого для MVP: примерно **4-7 рабочих дней** при фокусной разработке.

---

## 6. Критерии завершения реализации функционала

Функционал считается завершенным, когда:
- два пользователя стабильно общаются в одной комнате;
- третий пользователь не может войти в заполненную комнату;
- сообщения не попадают в другие комнаты;
- после выхода пользователей состояние комнаты удаляется;
- ключевые сценарии покрыты автоматическими тестами;
- CI зеленый, staging smoke-тест проходит успешно.
