# Стратегия тестирования

## Цель

Гарантировать, что приватный чат работает стабильно, комнаты изолированы,
а ограничения продукта (2 пользователя, no persistence) не нарушаются.

## Пирамида тестов

## 1) Unit (быстро, много)

Бэкенд:
- `RoomManager`:
  - создание комнаты
  - вход/выход
  - лимит 2 участника
  - удаление пустой комнаты

Фронтенд:
- хук `useChatSocket`:
  - обработка базовых событий
  - переходы между состояниями соединения
  - корректная фильтрация событий по `roomId`

## 2) Integration

Бэкенд WS:
- `join_room` -> `room_joined`
- второй вход -> `peer_joined`
- третий вход -> `room_full`
- `chat_message` доставляется только участникам комнаты
- disconnect -> `peer_left` + cleanup

Фронтенд:
- форма подключения в комнату
- отправка сообщения и рендер в списке
- отображение ошибок `room_full` и `error`

## 3) E2E (минимальный набор)

- Пользователь A создает комнату и отправляет ссылку
- Пользователь B подключается по ссылке
- A и B обмениваются сообщениями
- Пользователь C получает `room_full`

## Требования к PR

Минимум для каждой функциональной задачи:
- 1 тест на успешный сценарий
- 1 тест на ошибочный/краевой сценарий

Для изменения протокола:
- обновить backend integration тесты
- обновить frontend тесты обработки события
